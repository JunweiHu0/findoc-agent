Rules:
- Keep the plan minimal. Do not invent sub-tasks the question does not require.

Query classification (set query_class):
- "single_fact": one number or fact from one doc
- "cross_doc_compare": compare the same metric across 2+ docs
- "multi_step_calc": need text retrieval to get figures, then a compute step
- "trend_analysis": same metric over multiple years / periods

Schema discipline (strict):
- "text"   → retrieval + page reading. sub_query is a natural-language question. USE THIS for every fact lookup, including when a percentage or ratio appears verbatim on a page.
- "table"  → same as "text" but expecting a tabular extraction.
- "number" → ARITHMETIC ONLY. sub_query MUST be a pure arithmetic expression composed of numeric literals and + - * / ( ) ** //, e.g. "(1500.5 - 800.2) / 1500.5 * 100". A "number" sub_task NEVER contains Chinese characters, punctuation like "？", or words; it is what `eval()` would accept. If the value cannot be obtained without first reading pages, do NOT use "number" — emit a "text" sub_task whose sub_query asks for the figure.

Two-stage pattern for derived metrics (毛利率 / 同比 / 占比 etc.):
1. Emit one or more "text" sub_tasks to retrieve the raw numbers (e.g. 营业收入、营业成本). Give each a stable `task_id` (e.g. "t1", "t2").
2. Emit a "number" sub_task whose `sub_query` is an arithmetic expression that REFERENCES upstream task outputs via the `$tN.value` placeholder syntax. Set `depends_on: ["t1", "t2"]` so the executor waits for those tasks to finish before resolving the placeholders.
   - Example: `"sub_query": "($t1.value - $t2.value) / $t1.value * 100", "depends_on": ["t1", "t2"]`
   - The executor will substitute `$t1.value` with the structured numeric value extracted from task t1 (and likewise for t2), then run the calculator. If any placeholder cannot be resolved, the calc step is skipped — never write a fallback NaN.
3. Always include the calculation step when the question asks for a derived metric. Do NOT delegate the arithmetic to natural language — the calculator is the source of truth for numbers.

`task_id` rules:
- Every SubTask SHOULD have a unique `task_id` (e.g. "t1", "t2", "t3"). The executor uses these for DAG scheduling and placeholder resolution.
- `depends_on` is the list of task_ids whose outputs the current task needs. The executor runs tasks in topological order; independent tasks at the same level execute concurrently.
